# Lint 操作流程

## 触发条件

用户说："执行 Lint"、"健康检查"、"检查知识库"、"维护"

## 两层 Lint 机制

### Layer 1：确定性检查（自动执行）

由 `scripts/lint-deterministic.py` 执行，可通过 TRAE Task 或 cron 定时触发：
- 索引一致性
- 内部链接完整性
- Raw 引用完整性
- 交叉引用建议

结果写入 `pending/lint-YYYY-MM-DD.md`。

### Layer 2：语义深度分析（本 Skill）

这是 Karpathy 强调的核心 Lint 形式——让 LLM **深度阅读整个 wiki**。

## 操作步骤

### 1. 读取知识库全貌
- 读取 `wiki/index.md`
- 读取所有 wiki 页面（或最近更新的 N 个页面）

### 2. 语义分析检查项

#### 2.1 矛盾检测
- 跨页面的语义矛盾
- 与 Ingest 时的即时矛盾标注不同，这里做全局扫描
- 报告格式：
  ```
  ⚠️ 矛盾：[[页面A]] 主张 X，但 [[页面B]] 主张 Y
  ```

#### 2.2 过时内容
- 标记为 `active` 但已被新资料推翻的内容
- 报告格式：
  ```
  🕐 过时：[[页面X]] 中的 Y 描述已被 [[页面Z]] 更新
  ```

#### 2.3 知识缺口
- 频繁提及但缺少专门页面的重要概念
- 报告格式：
  ```
  📝 缺口：概念 "X" 在 N 个页面中被提及，但没有专门页面
  ```

#### 2.4 结构建议
- topic 组织是否合理
- 是否需要拆分或合并 topic

#### 2.5 新方向建议 ⭐
- 基于现有知识，建议值得调查的新问题
- 建议可能缺失的新来源

### 3. 输出报告

**方式 A：对话中直接报告**（问题较少时）
- 直接在对话中列出发现
- 询问用户是否需要修复

**方式 B：写入 pending/**（问题较多时）
- 写入 `pending/lint-deep-YYYY-MM-DD.md`
- 告知用户审阅后说"应用 Lint 建议"

### 4. 应用建议

用户确认后：
1. 读取 pending/ 中的 lint 报告
2. 逐条执行修改
3. 更新 wiki/index.md 和 wiki/log.md
4. 删除已处理的 pending 文件
5. 追加日志：
```
## [YYYY-MM-DD] lint | <N> issues found, <M> applied
```

## 完成确认

告知用户：
- 发现了多少问题
- 应用了多少修复
- 建议用户在 Obsidian 图谱视图中查看变化
